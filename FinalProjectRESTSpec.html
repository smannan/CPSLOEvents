General Points

The following design points apply across the document.

  1. All resource URLs are prefixed by some root URL, (e.g. http://www.softwareinventions.com/CHS/)
  2. All resources accept and provide only JSON body content. And per REST standards, all successful (200 code) DELETE actions return empty body.
  3. Some GET operations allow get-parameters. These are listed directly after the GET word. All get-parameters are optional unless given in bold.
  4. Absent documentation to the contrary, all DELETE calls, POST, and PUT calls with a non-200 HTTP response return as their body content, a list of JSON objects describing any errors that occured. Error objects are of form {tag: {errorTag}, params: {params}} where errorTag is a string tag identifying the error, and params is a possibly-empty array of additional values needed to fill in details about the error. E.g. {tag: "missingField", params: ["lastName"]}
  5. Resource documentation lists possible errors only when the error is not obvious from this General Points section. Relevant errors may appear in any order in the body. Missing field errors are checked first, and no further errors are reported if missing fields are found.
  6. All resource-creating POST calls return the newly created resource as a URI via the Location response header, not in the response body. The respose body for such POSTs is reserved for error information, per point 4.
  7. GET calls return one of the following. Response body is empty in the latter three cases. Get calls whose specified information is a list always return an array, even if it has just one or even zero elements.
  1. HTTP code OK and the specified information in the body.
  2. BAD_REQUEST and a list of error strings.
  3. UNAUTHORIZED for missing login.
  4. FORBIDDEN for insufficient authorization despite login
  5. NOT_FOUND for a URI that is not described in the REST spec if logged in, 401 if not.
  1. Fields of JSON content for POST and PUT calls are assumed to be strings, booleans, ints, or doubles without further documentation where obvious by their name or intent. In nonobvious cases, the docs give the type explicitly.
  2. All access requires authentication via login to establish the Authenticated User (AU); no resources are public except for Prss/POST (for initial registration), and Ssns/POST (to log in). Other resources may be restricted based on admin status of AU. The default restriction is to allow only access relevant to the AU, unless the AU is admin, in which case access to any Person's info is allowed.
  3. Any database query failure constitutes a server error (status 500) with a body giving the error object returned from the query. Ideally, no request, however badly framed, should result in such an error except as described in point 11
  4. The REST interface does no general checking for forbiddenField errors, unless the spec specifically indicates it will. Absent such checking, non-specified body fields in PUT/POST calls may result in database query errors and a 500 code, as may an empty body when body content is expected.
  5. Required fields may not be passed as null, undefined or "". Doing so has the same outcome as if the field were entirely missing
  6. All times are integer values, in mS since epoch.
  7. Non JSON parseable bodies result in 500 error.

Error Codes

The possible error codes, and any parameters, are as follows.

missingField Field missing from request. Params[0] gives field name

badValue Field has bad value. Params[0] gives field name

notFound Entity not present in DB -- for cases where an Event, Person, etc. is not there.

badLogin Email/password combination invalid, for errors logging.

dupEmail Email duplicates an existing email

noTerms Acceptance of terms is required

noOldPwd Change of password requires an old password

oldPwdMismatch Old password that was provided is incorrect.

dupTitle Event title duplicates an existing one

dupEnrollment Duplicate enrollment

queryFailed Query failed (server problem)

Resources for User Management, including Registration

Prss

Collection of all current students or other users.

GET email={email or email prefix}

Returns list of zero or more Persons. Limits response to Persons with specified email or email prefix, if applicable. No data for other than the AU is returned in any event. This may result in an empty list if e.g. a user asks for an email not their own. Data per person:

email principal string identifier, unique across all Persons

id id of person with said email, so that URI would be Prss/{id}

POST

Adds a new Person. No AU required, as this resource/verb is used for registration, but an AU is allowed, and an admin AU gets special treatment as indicated.

email unique Email for new person

firstName

lastName

password

termsAccepted boolean--were site terms and conditions accepted?

City

State

Zip code

Country

Email and lastName required and must be nonempty. Error if email is nonunique. Error if terms were not accepted. Nonempty password required. City, State, Zip code, and Country may be left empty.

Prss/{prsId}

GET

Returns array with one element for Person {prsId}, with fields as specified in POST for Prss, plus dates termsAccepted and whenRegistered, less password. The dates give time of term acceptance and registration, and will generally be equal, but are listed separately for legal reasons. AU must be person {prsId}.

PUT

Update Person {prsId}, with body giving an object with one or more of firstName, lastName, password, role. Attempt to change other fields in Person such as termsAccepted or whenRegistered results in BAD_REQUEST and forbiddenField error(s). All changes require the AU be the Person in question. An additional field oldPassword is required for changing password. Password, if supplied, must be nonempty.

DELETE

Delete the Person in question, including all Events and Reservations owned by Person. Require AU to be the person in question.

Prss/{prsId}/Rsvs

GET

AU must be person in question. Return an array of 0 or more elements, with one element for each Reservation in the system belonging to the AU.

id Id of the reservation

firstName first name of the person

lastName last name of the person

status Person’s reservation status

eventId Id of the event this reservation

Title Name of the event

Date Time and date of the event in ms since the epoch

Prss/{prsId}/Rsvs/{rsvId}

PUT

Update the Reservation of {rsvId}. AU must be owner of the reservation. Error badValue if status is invalid.

status Status of the attendance

Ssns

Login sessions (Ssns) establish an AU and are required for most service access. A user obtains one via POST to Ssns.

POST

A successful POST generates a browser-session cookie that will permit continued access for 2 hours. Indicated Person becomes the AU. An unsuccesful POST results in a 400 with a badLogin tag and no further information.

email Email of user requesting login

password Password of user

Ssns/{cookie}

GET

Returns, for the indicated session, a single object with same properties as one element of the array returned from Ssns GET. AU must be owner of session.

DELETE

Log out the specified Session. AU must be owner of Session.

Resources for Event

The following resources allow creation, deletion, and management of Events. Any AU may GET information on a public event. Events that are private require the AU to be on the reservations list. Any AU may POST an event. Only AU that have been invited to an event may POST to reservations.

Evts

GET start=<datetime>end=<datetime>loc=<zipcode>owner=<prsId>

Any AU is acceptable, though some login is required. Return an array of 0 or more elements, with one element for each Event in the system, limited to Events after the start date and before the end date or with the same zip code as location if query param is given. If the owner query param is provided, only events that the AU has created will be returned. Only Events that are public or the AU has been invited to will be returned:

id Id of the Event

title Name of the Event

orgId Organizer of the Event

date Datetime of the Event, in ms since epoch

City City of the Event

State State of the Event

Country Country of the Event

Address Address of the Event

Private Whether or not the event is private

desc Description of the Event, limited to 500 chars

POST

Any AU is acceptable, though some login is required. Create a new Event, owned by the current AU. Error dupTitle if title is a duplicate. Error badValue if private is not 0 or 1. Error badValue if date is negative or before the current date. Error badValue  if any of the character limits are exceeded.

name Name of the new Event, limited to 80 chars.

city City of the Event, limited to 50 chars

state State of the Event, limited to 50 chars

country Country of the Event, limited to 50 chars

address Street Address of the Event, limited to 50 chars

datetime Datetime of the Event

desc Description of the Event, limited to 500 chars

private Boolean value, if event is private or not.

Evts/{evtId}

GET

Return single object having same properties as one of the array elements returned by Evts GET, for just the indicated Evt. AU must be invited to the event or the event must be public. Error not found if event does not exist or has been deleted.

PUT

Update the Event. Fields allowed are as is in POST Evts. Error dupTitle if title is duplicate. AU must be Event owner or admin.

DELETE

Delete the Event, including all associated Reservations. AU must be Event organizer. 

Evts/{evtId}/Rsvs

GET

Return an array of 0 or more elements, with one element for each Reservation in the system of the form {firstName, lastName, status}. AU can see all the results if {evtId} is private and the user is invited, or if the event is public. May return an empty array if AU tries to GET to an event that is private or they are not invited to. Some AU is required.

FirstName First name of the user

LastName Last name of the user

status Status of the user’s attendance to the event. For example, Going, Maybe, Not Going

POST

A POST to Rsvs will add a user to the Reservations List of the event. This allows Events to be shared by the AU. Any AU can POST to public Events to RSVP. For a private event, AU must be the event organizer to POST. Unless specified by the AU status will default to not going. Error not found if personId or eventId does not exist. Error badValue if status is anything other than Going, Maybe, Not Going. Error missingField if personId or eventId are missing.

personId Id of the person to make the reservation for

status Status of the attendance to the event. For example, Going, Maybe, Not Going

Evts/{evtId}/Rsvs/{rsvId}

DELETE

Remove the Reservation of {rsvId}. AU must be Event organizer or reservation owner to delete the reservation. This is useful for removing a person from the invited list of a private event.
